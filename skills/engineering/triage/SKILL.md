# Triage

Move GitHub issues through a small state machine of triage roles via `gh`.

**Conventions** (see top-level README): `gh` CLI only; the canonical roles below are the source of truth.

Every comment or issue posted during triage **must** start with this disclaimer:

```
> *This was generated by AI during triage.*
```

## Reference docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — how to write durable agent briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — how the `.out-of-scope/` knowledge base works

## Roles

Two **category** roles:

- `bug` — something is broken
- `enhancement` — new feature or improvement

Five **state** roles:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter for more information
- `ready-for-agent` — fully specified, ready for an AFK agent
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Every triaged issue should carry exactly one category role and one state role. If state roles conflict, flag it and ask the maintainer before doing anything else.

State transitions: an unlabeled issue normally goes to `needs-triage` first; from there it moves to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the reporter replies. The maintainer can override at any time — flag transitions that look unusual and ask before proceeding.

If the project's actual labels diverge from these canonical names (e.g. `triage:needs-info` instead of `needs-info`), check with `gh label list` once at the start and map accordingly.

## Invocation

The maintainer invokes `/triage` and describes what they want in natural language. Examples:

- "Show me anything that needs my attention"
- "Let's look at #42"
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## Show what needs attention

Query GitHub via `gh` and present three buckets, oldest first:

```bash
# Unlabeled — never triaged
gh issue list --search "no:label" --limit 50 --json number,title,createdAt

# needs-triage in progress
gh issue list --label "needs-triage" --limit 50 --json number,title,updatedAt

# needs-info with reporter activity since the last triage notes
gh issue list --label "needs-info" --limit 50 --json number,title,updatedAt,comments
```

Show counts and a one-line summary per issue. Let the maintainer pick.

## Triage a specific issue

1. **Gather context.**
   ```bash
   gh issue view <N> --comments
   ```
   Read body, comments, labels, reporter, dates. Parse any prior triage notes so you don't re-ask resolved questions. Explore the codebase using the project's domain glossary, respecting ADRs in the area. Read `.out-of-scope/*.md` and surface any prior rejection that resembles this issue.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the issue. Wait for direction.

3. **Reproduce (bugs only).** Before any grilling, attempt reproduction: read the reporter's steps, trace the relevant code, run tests or commands. Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much stronger agent brief.

4. **Grill (if needed).** If the issue needs fleshing out, run a `/grill-with-docs` session.

5. **Apply the outcome:**

   ```bash
   # ready-for-agent — post the agent brief, swap labels
   gh issue comment <N> --body "$(cat <<'EOF'
   > *This was generated by AI during triage.*

   <agent brief per AGENT-BRIEF.md>
   EOF
   )"
   gh issue edit <N> --remove-label "needs-triage,needs-info" --add-label "ready-for-agent,bug"  # or enhancement

   # ready-for-human — same brief, note why it can't be delegated
   gh issue edit <N> --remove-label "needs-triage,needs-info" --add-label "ready-for-human,bug"

   # needs-info — post triage notes
   gh issue comment <N> --body "$(cat <<'EOF'
   > *This was generated by AI during triage.*

   ## Triage Notes

   **What we've established so far:**
   - point 1
   - point 2

   **What we still need from you (@reporter):**
   - question 1
   - question 2
   EOF
   )"
   gh issue edit <N> --remove-label "needs-triage" --add-label "needs-info"

   # wontfix (bug) — polite explanation, then close
   gh issue comment <N> --body "<reasoning>"
   gh issue close <N> --reason "not planned" --add-label "wontfix"

   # wontfix (enhancement) — write to .out-of-scope/, link, close
   #   create .out-of-scope/<slug>.md per OUT-OF-SCOPE.md
   gh issue comment <N> --body "Recorded in .out-of-scope/<slug>.md — <link>"
   gh issue close <N> --reason "not planned" --add-label "wontfix"
   ```

   Capture everything resolved during grilling under "What we've established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info".

## Quick state override

If the maintainer says "move #42 to ready-for-agent", trust them and apply the role directly. Confirm what you're about to do (label changes, comment, close), then act. Skip grilling. If moving to `ready-for-agent` without a grilling session, ask whether they want an agent brief.

## Resuming a previous session

If prior triage notes exist on the issue, read them, check whether the reporter has answered any outstanding questions, and present an updated picture before continuing. Don't re-ask resolved questions.
